LangChain이 새로 공개한 **LangSmith Agent Builder**는 개발자만 쓰던 에이전트 세계를 일반 업무 담당자에게도 열어주는 도구입니다. 코드 한 줄 없이, 대화로 에이전트를 만들고 메모리·도구·트리거까지 붙입니다. 이번 글에서는 이 도구가 기존 노코드 자동화와 무엇이 다른지, 어떻게 동작하는지, 그리고 실무에서 어디부터 써볼 만한지 정리해 봅니다.

## 핵심 요약

- LangChain이 **LangSmith Agent Builder**를 프라이빗 프리뷰로 공개했습니다(2025년 10월 29일 발표, 저자 Brace Sproul · Sam Crowder).
- 비주얼 워크플로 빌더가 아니라 **에이전트 빌더**입니다. 분기 노드를 일일이 잇는 게 아니라, LLM이 상황에 맞춰 의사결정을 위임받습니다.
- 에이전트는 네 가지 구성요소로 짭니다 — **프롬프트, 도구(MCP), 트리거, 서브에이전트**.
- 빈 캔버스 대신 **대화로 시작**합니다. 시스템이 후속 질문을 던지며 프롬프트를 자동 생성해 줍니다.
- 메모리가 내장돼 있어, 한 번 교정하면 다음부터는 그 교정을 기억합니다. 내부에서는 LangChain의 **deepagents** 패키지가 엔진 역할을 합니다.

## 비주얼 워크플로가 아닙니다 — 에이전트 빌더입니다

기존 노코드 자동화 도구는 대체로 박스를 잇는 방식이었습니다. 화면은 직관적으로 보여도 흐름이 복잡해지면 분기 노드와 조건식이 거미줄처럼 얽혀 관리가 점점 어려워집니다. LangChain 팀은 이 한계를 두 가지로 정리합니다.

> "비주얼 워크플로 빌더는 사실 '낮은 진입 장벽'이 아닙니다. 그리고 복잡한 일은 비주얼 빌더로 다루기에 너무 빨리 복잡해집니다."

LangSmith Agent Builder는 그 박스 잇기를 LLM에게 맡깁니다. 정해둔 경로를 따라가는 대신, 에이전트가 상황을 보고 그때그때 판단하도록 둔다는 뜻입니다. 그래서 더 많은 사용자가 다룰 수 있고, 단순 자동화를 넘어 복잡한 업무까지 들고 갈 수 있다는 게 이번 발표의 핵심 논점입니다.

![LangSmith Agent Builder UI](/images/langsmith-no-code-agent-builder/source-screenshot.png)

## 에이전트를 구성하는 네 가지 요소

LangSmith 안의 모든 에이전트는 다음 네 가지 부품으로 만듭니다.

### 1. 프롬프트(Prompt) — 에이전트의 뇌

에이전트가 무엇을 하고 무엇을 하지 말아야 하는지를 적어둔 본문입니다. 비주얼 워크플로의 복잡함을 모두 프롬프트로 밀어 넣은 셈인데, 잘 쓴 프롬프트가 곧 잘 만든 에이전트입니다. 그래서 LangChain은 이 작업을 사람이 혼자 짜내지 않도록 도구를 따로 붙였습니다(아래에서 다시 다룹니다).

### 2. 도구(Tools) — MCP로 바깥 세상에 손을 뻗다

에이전트가 메일을 보내고, 일정을 잡고, 티켓을 만드는 일은 모두 **MCP(Model Context Protocol)** 로 연결합니다. LangSmith에는 기본 제공 도구가 있고, 직접 만든 MCP 서버도 손쉽게 붙일 수 있습니다. 이번 발표에서 같이 공개된 **Agent Authorization** 기능 덕분에 Gmail, Slack, LinkedIn, Linear 같은 사내 승인 도구를 보안 정책 안에서 연결할 수 있습니다.

### 3. 트리거(Triggers) — 에이전트를 깨우는 신호

에이전트는 채팅 입력만 받는 게 아닙니다. 메일이 도착했을 때, 특정 슬랙 채널에 메시지가 올라왔을 때, 혹은 정해진 시간에 — 트리거를 걸어두면 에이전트가 알아서 깨어나 일을 처리합니다.

### 4. 서브에이전트(Subagents) — 복잡해지면 쪼개기

처음에는 프롬프트 한 장에 다 담는 걸 권장합니다. 그런데 일이 복잡해지면 한 에이전트 안에 모든 책임을 욱여넣는 게 오히려 부담이 됩니다. 이때 특정 업무에 집중한 작은 서브에이전트를 만들어 분담시키는 식으로 시스템을 가볍게 유지합니다.

## 빈 화면이 아니라 대화로 시작합니다

에이전트 빌더에서 가장 까다로운 일은 사실 **좋은 프롬프트를 쓰는 것**입니다. 그런데 대다수 사용자에게는 프롬프트 엔지니어링 경험이 없습니다. 게다가 한 번 잘 써둔 프롬프트도, 새로운 상황을 만나면 다시 손봐야 합니다.

LangSmith Agent Builder는 이 두 문제를 다음과 같이 풉니다.

- **대화로 시작합니다.** "어떤 일을 시키고 싶나요?"라는 첫 질문에 평소 쓰는 말투로 답하면 됩니다. 그러면 시스템이 후속 질문을 던지며 빈 칸을 채워 줍니다. 어떤 도구를 쓸지, 어떤 트리거가 필요한지도 대화 속에서 자연스럽게 정해집니다.
- **시간이 지나도 기억합니다.** 에이전트는 프롬프트뿐 아니라 사용하는 도구에 대한 메모리도 내장하고 있습니다. 잘못된 응답을 교정해 주면 그 교정을 기록해 두고, 다음부터는 같은 실수를 반복하지 않습니다. 매번 같은 잔소리를 되풀이할 필요가 없습니다.

## 실제로 어디에 쓸 만한가

LangChain은 사내 생산성 시나리오를 가장 먼저 권합니다.

- **하루 일정 요약 + 회의 준비 자료 만들기.** 아침에 메일 한 통으로 오늘의 일정과 각 미팅에 필요한 배경 자료를 받습니다.
- **메일 처리 에이전트.** 들어온 메일을 읽고, 답장 초안을 쓰거나, Linear 티켓을 만들거나, 슬랙으로 관련자에게 핑을 보냅니다. 외부로 나가는 메시지는 사람이 한 번 더 확인하고 보낼 수 있도록 승인 단계를 끼워 넣을 수 있습니다.
- **Salesforce·세일즈 보조.** 영업 도구와 연결해 고객 응대를 보조하는 에이전트도 같은 흐름으로 만들 수 있습니다.

핵심은 **사람이 매일 반복하는 사무 업무**, 그리고 **승인이 필요한 정도의 책임을 위임할 수 있는 업무**부터 시작하라는 겁니다.

## 내부 구조 — deepagents가 엔진입니다

LangSmith Agent Builder는 그냥 새로 만든 노코드 도구가 아닙니다. LangChain이 3년간 LangChain·LangGraph로 쌓은 경험, 그리고 초기 실험작인 Open Agent Platform에서 배운 것을 다시 정리해 올린 결과물입니다.

엔진 역할을 하는 것은 LangChain의 **deepagents** 패키지입니다. deepagents는 에이전트에 다음과 같은 기능을 줍니다.

- **계획 수립(planning)** — 복잡한 요청을 작은 단계로 쪼개기
- **지속 메모리(persistent memory)** — 세션이 끝나도 살아남는 상태 보관
- **서브태스크 분할** — 큰 문제를 작은 작업으로 위임

덕분에 사용자가 모든 분기 경로를 미리 그려둘 필요 없이, 에이전트가 실시간으로 문제를 풀어 나갑니다. 비주얼 워크플로 시절에는 시나리오를 빠짐없이 그려야 한다는 부담이 있었는데, 그 부담을 LLM에게 넘긴다는 점이 가장 큰 차이입니다.

기존 LangChain 생태계 사용자라면, 오픈소스 LangChain·LangGraph와 LangSmith Agent Builder를 언제 어떻게 나눠 쓸지에 대한 안내도 함께 공개됐습니다. 코드로 정교하게 짜야 할 사용자용 핵심 에이전트는 기존 프레임워크로, 사내 업무 자동화처럼 빠르게 만들고 손볼 일이 많은 에이전트는 Agent Builder로 — 정도가 1차 가이드라인입니다.

## 실무자가 볼 핵심 포인트

- **노코드의 정의를 다시 합니다.** 박스를 잇는 노코드가 아니라, 프롬프트를 대화로 만드는 노코드입니다. 비기술 사용자도 진짜로 만들 수 있는 수준까지 진입 장벽을 내렸습니다.
- **MCP가 표준 연결 통로입니다.** 사내 시스템에 연결하려면 MCP 서버를 갖춰두는 쪽이 유리합니다. Gmail·Slack·LinkedIn·Linear는 Agent Authorization으로 기본 지원됩니다.
- **메모리가 운영을 바꿉니다.** "한 번 알려준 건 다시 안 알려줘도 된다"는 단순한 약속이지만, 사내 도구 운영에서는 상당한 차이를 만듭니다. 도입 시 어떤 교정 기록을 남길지 정책을 같이 정해두는 편이 좋습니다.
- **승인 흐름은 처음부터 설계합니다.** 메일 전송, 외부 메시지, 티켓 생성 같은 행동에는 사람의 최종 승인을 끼워 넣는 흐름이 권장됩니다. 도입 초기에는 더더욱 그렇습니다.
- **서브에이전트는 나중에.** 처음부터 잘게 쪼개려 하지 말고, 한 에이전트의 프롬프트에 모두 담아 보다가 관리가 어려워질 때 분리하면 됩니다.
- **deepagents를 기억해 둡니다.** Agent Builder가 어떻게 복잡한 일을 다루는지 이해하려면, 결국 그 엔진인 deepagents의 계획·메모리·서브태스크 모델을 알아두는 게 도움이 됩니다.

LangSmith Agent Builder는 현재 프라이빗 프리뷰이고, LangChain은 커뮤니티 피드백을 받으며 기능을 넓혀 갈 계획입니다. 사내에서 반복 사무 업무가 많은 팀이라면, 메일 트리거 한 건짜리 작은 에이전트부터 시작해 보면 감이 가장 빨리 옵니다.
