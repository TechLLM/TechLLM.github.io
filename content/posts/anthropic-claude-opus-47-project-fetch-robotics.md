## 핵심 요약

- Anthropic이 2025년 8월에 진행했던 **Project Fetch**를 새 Claude 모델로 다시 돌렸습니다.
- 실험 도구는 **Claude Code 안에서 동작하는 Claude Opus 4.7**, 사람은 노트북을 로봇에 연결하고 초기 프롬프트만 던지는 역할에 머물렀습니다.
- Opus 4.7은 사람이 1년 전에 풀었던 일부 과제를 **가장 빠른 인간 팀보다 약 20배 빠르게** 끝냈고, 단계별로도 **최소 10배 이상** 빨랐습니다.
- 다만 비치볼을 정확히 굴려 시작 지점으로 되돌리는 **"가져오기(fetching)" 동작**에서는 여전히 막혔습니다. 정밀 모션 제어는 미해결입니다.
- Anthropic은 "이건 로보틱스 돌파구가 아니다"라고 못 박았습니다. 발전의 원인은 로보틱스 특화가 아니라 **모델 전반의 스케일업** 때문이라는 설명입니다.

## 개요

Anthropic이 자체 블로그에 올린 두 번째 Project Fetch 보고서는 짧지만 메시지가 분명합니다. 같은 로봇, 같은 과제, 다른 결과. 1년 사이에 사람을 따라잡던 LLM이 사람을 한참 앞질러 버렸습니다. 그러면서도 "지금까지 LLM이 로보틱스를 풀었다고 보기는 어렵다"라고 스스로 선을 그었습니다. 이 둘이 같이 적힌 글이라는 점이 흥미롭습니다.

## Project Fetch가 뭐였나

Project Fetch는 2025년 8월에 처음 진행된 실험입니다. Anthropic 직원들을 두 팀으로 나눠 한쪽은 Claude의 도움을 받으며 로봇을 다뤘고, 다른 한쪽은 웹 검색과 머릿속에 든 지식만으로 같은 과제를 풀어야 했습니다. 과제는 단순한 로봇 조작이 아니라 **센서와 어떻게 통신할지, 어떤 인터페이스로 모터를 움직일지, 비치볼을 어떻게 다룰지**를 직접 설계하는 쪽에 가까웠습니다.

이번 재실험에서는 사람의 비중을 더 줄였습니다. 연구자가 한 일은 세 가지뿐입니다.

1. 노트북을 로봇에 연결합니다.
2. 시작 프롬프트를 한 줄 입력합니다.
3. Claude가 제안하는 명령과 과제 전환을 승인합니다.

나머지는 Claude Code 위에서 Opus 4.7이 알아서 합니다. 어떤 라이브러리를 쓸지, 센서 신호를 어떻게 해석할지, 잘못된 명령을 어떻게 고칠지를 모두 직접 결정합니다.

## 무엇이 빨라졌나

Anthropic이 강조한 숫자는 두 가지입니다.

- **약 20배** — Opus 4.7이 사람 손 없이도, 가장 빨랐던 인간 팀이 풀었던 과제들을 묶어서 약 20배 빠르게 마쳤습니다.
- **최소 10배** — 인간 팀 중 한 팀이라도 통과했던 단계라면, 같은 단계를 Opus 4.7은 적어도 10배 빠르게 통과했습니다.

속도만 빠른 게 아닙니다. Anthropic은 모델이 **사람들이 2025년에 고민하던 선택지를 거의 즉시 결정했다**고 적었습니다. 예를 들어 로봇 센서와 어떤 방식으로 연결할지 같은 문제입니다. 사람은 며칠 고민하던 문제를 모델은 몇 분 안에 골라냈고, 짠 코드 상당수는 **처음 돌릴 때부터 그대로 동작**했습니다. Claude의 도움을 받았던 사람 팀과 비교해도 작성한 코드 양이 훨씬 적었습니다. 적은 코드로 같거나 더 나은 결과가 나온 셈입니다.

## "그래도 로보틱스를 푼 건 아니다"

Anthropic이 흥미로운 지점은 자기 결과를 스스로 식히는 부분입니다. 글에는 이렇게 적혀 있습니다.

> "LLM이 이제 로보틱스를 풀었다는 뜻은 아니다. 전혀 그렇지 않다."

Opus 4.7은 **비치볼을 정확히 굴려 시작 지점으로 되돌리는 동작**에서 막혔습니다. 비치볼을 밀 자세까지는 로봇을 끌고 갔지만, 환경 피드백을 보면서 모션을 정교하게 보정하는 부분에서 실패한 겁니다. 즉, 계획 단계는 모델이 사람을 앞질렀지만, **연속적인 모션 제어**에는 여전히 사람 손이 필요한 상태입니다.

또 하나, Anthropic은 이 성능 향상이 **로보틱스 전용 학습 덕분이 아니라**고 못 박았습니다. 로봇 데이터를 더 모은 것도 아니고, 로보틱스에 맞춘 RLHF를 한 것도 아닙니다. 그저 **일반 모델 능력 자체가 커졌고**, 그 결과 로봇 계획 같은 일도 같이 풀린 겁니다.

이 점이 의외로 무겁습니다. Anthropic이 글 안에서 정리한 흐름은 이렇습니다.

> 처음에는 모델이 사람을 돕고, 다음에는 사람이 모델을 돕고, 결국에는 모델이 점점 더 많은 일을 혼자 끝낸다.

지난 1년 사이에 로봇 계획이라는 영역에서 이 단계가 한 칸씩 미끄러져 내려갔습니다.

## 실무자가 볼 핵심 포인트

- **모델 스케일업이 그대로 도메인 능력으로 번진다**는 메시지가 다시 한번 확인됐습니다. 도메인 특화 학습 없이도 일반 모델 향상이 로봇 계획 단계를 빠르게 만들었습니다. 자체 도메인 모델을 따로 만들 계획이 있다면, 일반 LLM의 진척 속도와 비교해 다시 검토해 볼 만합니다.
- **계획 vs 제어가 분리된다.** Opus 4.7은 어떤 라이브러리를 쓸지, 어떤 흐름으로 작업을 분해할지는 빠르게 결정했지만, 연속 모션 제어는 여전히 실패했습니다. 로봇 스택을 짤 때 **상위 의사결정은 LLM에 맡기고, 정밀 제어 루프는 별도 컨트롤러에 두는 분리 설계**가 한동안 더 유효해 보입니다.
- **사람의 역할이 "감독"으로 좁아진다.** 이번 실험에서 사람의 일은 연결과 승인이 거의 전부였습니다. 사내에서 LLM 기반 자동화를 도입할 때, 사람이 어디서 승인을 줘야 하는지, 어떤 기준으로 중단할지를 명확히 설계해야 안전과 책임 라인이 흐려지지 않습니다.
- **코드를 적게 짜고도 같은 결과**를 낸다는 점은 비용 측면에서도 의미가 큽니다. 모델이 짧은 코드로 한 번에 맞추기 시작하면, 사후 디버깅과 코드 리뷰 비용도 같이 줄어듭니다. 다만 "한 번에 도는 코드"라도 안전성 검증은 자동화해 두는 편이 좋습니다.
- **하드웨어 쪽에서도 같은 곡선을 배제하지 말라**는 Anthropic의 경고는 흘려듣기 어렵습니다. "모델이 자기 소프트웨어 도구를 직접 만든다"는 일은 몇 년 전만 해도 황당하게 들렸지만 지금은 일상이 됐고, 같은 일이 하드웨어 영역에서도 벌어질 수 있다고 회사는 적었습니다.
