## 개요

코드 에디터 커서(Cursor)가 자체 행사에서 새 소식 세 개를 한꺼번에 풀었습니다. 첫째는 **자체 학습한 첫 AI 모델**, 둘째는 **새 깃 플랫폼 Origin**, 셋째는 **iOS 베타 Cursor Mobile**입니다. 세 발표의 공통점은 분명합니다. 코딩 도구를 사람만 쓰는 게 아니라 **사람과 에이전트가 같이 쓰는 환경**으로 바꾸겠다는 신호죠.

원문에 따르면 커서의 모회사 **Anysphere가 최근 SpaceX에 인수됐고**, 자체 모델 학습도 그 SpaceX와의 협업으로 진행 중이라고 합니다. 이 사실 자체는 이전에 공개돼 있었고, 이번 행사에서 새로 더해진 건 **출시 시점과 모델 스펙**이었습니다.

## 핵심 요약

- **자체 AI 모델**: 외부 오픈소스 베이스 없이 처음부터 학습. 출시는 *몇 주 안*.
- **모델 규모**: 오푸스(Opus)·GPT급. 이전 커서 모델보다 **연산량 10~20배**.
- **용도**: 코딩에만 묶지 않고 **그 너머의 작업**까지 노림.
- **Origin**: 사람과 AI 에이전트를 같이 받아주는 새 깃 플랫폼. **수천 개 에이전트의 동시 읽기·쓰기**를 부하 테스트로 검증.
- **Origin 기능**: 머지 충돌 해소, CI 실패 자동 수정, 코멘트 처리. 현재 사내·일부 파트너 운영, **일반 공개는 올가을 목표**.
- **Cursor Mobile**: iOS 베타. **막힌 에이전트 풀기, 스크린샷 리뷰, 로컬 에이전트 원격 제어**까지.

## 본문

### 자체 모델 — 코딩 너머를 노리는 첫 번째 자기 모델

공동 창업자 마이클(Michael)이 발표 자리에서 말한 핵심은 두 가지입니다. **지금 학습이 돌고 있다**는 점, 그리고 **몇 주 안에 배포한다**는 점입니다. 그동안 커서가 내놓던 Composer 계열은 오픈소스 모델을 베이스로 후속 학습을 얹은 형태였는데, 이번 모델은 **밑바닥부터(scratch) 새로 학습**합니다. 다시 말해 같은 회사의 모델이라도 성격이 좀 다른 셈입니다.

크기는 **Opus, GPT 계열과 견줄 만한 규모**라고 했습니다. 이전 커서 모델 대비 **연산량 10~20배**라는 표현이 같이 나왔습니다. 모델 카드가 아직 없으니 정확한 파라미터·토큰 수치는 베일에 있지만, 적어도 *그동안 커서가 다루던 크기와는 다른 체급*에 들어간다는 얘기입니다.

흥미로운 점은 용도입니다. 커서는 "**코딩에만 쓰려고 만든 모델이 아니다**"라고 했습니다. 코드 에디터 회사가 자체 모델을 만든다고 했을 때 가장 먼저 떠오르는 자리는 자동완성·리팩터링 같은 IDE 안쪽인데, 거기서 한 발 더 나가겠다는 뜻입니다.

또 하나, 이 학습은 처음부터 **SpaceX와의 공동 작업**으로 알려져 있었습니다. 이번 발표의 새 정보는 그 협업의 산물이 **드디어 사용자 손에 닿는 시점**이 가까워졌다는 사실입니다.

### Origin — 사람과 에이전트가 같이 쓰는 깃

두 번째 발표는 **Origin**, 새 깃 플랫폼입니다. 공동 창업자 토마스(Tomas)가 소개했는데, 그는 커서가 **Graphite를 인수**하면서 합류한 인물입니다. 깃 인프라 쪽을 길게 다뤄온 사람이 가져온 결과물이라는 점에서 무게가 있습니다.

Origin은 기존 깃 위에 무언가를 얹는 도구가 아니라, **클라우드 위에 새 아키텍처로 다시 짠 깃**이라고 합니다. 왜 다시 짰을까요. 이유는 부하 테스트 결과에 그대로 나옵니다. 팀이 시뮬레이션한 시나리오는 **수천 개의 에이전트가 같은 저장소를 동시에 읽고 쓰는** 상황입니다. 사람만 쓸 때는 거의 일어나지 않는 경합이죠.

기능 면에서 Origin은 단순히 저장소를 보관해 주는 게 아닙니다.

- **머지 충돌**을 자동으로 풀어주고,
- **실패한 CI 테스트**를 직접 고치고,
- 코드 리뷰에서 달리는 **코멘트**까지 받아 처리합니다.

지금은 커서 사내와 일부 파트너에서 돌고 있고, **일반 공개는 올가을(fall)** 목표라고 못 박았습니다. 깃허브·깃랩이 자리를 단단히 잡고 있는 시장에 다시 깃 플랫폼을 들고 들어온다는 건 보통 일이 아닙니다. 다만 "에이전트가 PR 다섯 개씩 동시에 던지는" 흐름이 보편화된다면, **사람 기준으로 설계된 저장소 인프라는 한 번 더 흔들리게** 될 가능성도 있죠.

### Cursor Mobile — 막힌 에이전트를 휴대폰으로 푸는 앱

세 번째는 **Cursor Mobile**, iOS 베타입니다. 발표에서 강조한 그림은 이렇습니다. 데스크톱에서 에이전트를 돌려두고 자리를 비웠는데, 도중에 에이전트가 막혔다고 칩시다. 지금까지는 자리에 돌아올 때까지 그 작업이 그대로 멈춰 있어야 했습니다.

Cursor Mobile은 그 상황을 휴대폰에서 풀게 해줍니다.

- **원격으로 에이전트 관리**
- **막힌 작업 풀어주기(unblock)**
- 에이전트가 만들어낸 **스크린샷 검토와 코멘트**
- 로컬 머신에서 돌아가는 에이전트의 **원격 제어**

마지막 항목이 특히 눈에 띕니다. 클라우드 에이전트를 부르는 게 아니라, **내 맥북에서 돌고 있는 에이전트**를 휴대폰으로 잡아 쥐고 조작한다는 의미입니다. 발표 그대로라면 베타 단계임에도 활용 범위가 꽤 구체적입니다.

### 정리하면

세 발표를 따로 보면 각각의 기능이지만, 같이 보면 그림이 하나입니다. **모델은 커서가 직접 만들고, 저장소도 커서가 짠 인프라에 두고, 컨트롤은 휴대폰에서도 가능하게** — 코딩 도구의 경계가 점점 IDE 바깥으로 밀려나는 모양새입니다. 단, 자체 모델은 아직 배포 전이고, Origin은 일반 공개 전, Cursor Mobile도 베타라는 점은 분명히 해둘 필요가 있습니다. 발표 시점과 실제 손에 잡히는 시점 사이의 간격은 늘 한 박자 있죠.

## 실무자가 볼 핵심 포인트

- **모델 평가는 출시 후로**: "Opus·GPT급, 연산 10~20배" 같은 표현은 회사 발표 기준입니다. 실제 코드 작업·에이전트 시나리오에서의 성능은 출시 뒤 자체 벤치마크로 확인해야 의미가 있습니다.
- **Origin은 깃허브 대체가 아니라 *에이전트 워크로드 대응*으로 보자**: "사람 워크플로 + 수천 에이전트 동시 쓰기"가 핵심 시나리오입니다. 단일 개발자가 평소에 깃허브로 충분하다면 당장은 도구 교체 이슈가 아닙니다. 다만 사내에서 에이전트 PR 동시 운영을 시도 중이라면, 올가을 일반 공개 시점에 **머지 충돌 자동화 / CI 자동 수정** 두 항목은 검토 가치가 있습니다.
- **Cursor Mobile은 *원격 제어*가 핵심**: 모바일에서 코딩하라는 발표가 아닙니다. "에이전트가 자율로 돌다가 멈출 때 사람이 짧게 개입한다"는 워크플로 전제가 강해진다는 신호입니다.
- **SpaceX·Anysphere 구도 주시**: 모델 학습 인프라가 SpaceX 쪽에 묶여 있다는 점은, 향후 가격·가용성·계약 조건에 영향을 줄 수 있는 변수입니다. 도구 선택 기준에 *벤더 종속*도 한 줄 추가해둘 만합니다.
