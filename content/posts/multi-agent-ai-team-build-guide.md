## 개요

AI 에이전트 하나는 강력하다. 그런데 에이전트 팀이 함께 작동하면 차원이 다른 이야기가 된다.

2026년 5월 6일, Anthropic은 Code with Claude 행사에서 Claude Managed Agents의 멀티에이전트 오케스트레이션을 발표했다. 이제 단일 작업에 최대 20개의 전문화된 에이전트를 **동시에** 병렬로 실행할 수 있다. 순서대로가 아니라, 진짜 동시에.

Netflix는 수백 개의 빌드 로그를 동시 분석하는 데 이 구조를 쓴다. Harvey(법률 AI)는 복잡한 법률 문서 처리에, Shopify는 2026년 3분기까지 자율 코딩 90%를 목표로 이미 도입 중이다. 실험이 아니라 이미 프로덕션이다.

## 핵심 요약

- **멀티에이전트의 진짜 강점은 속도가 아니라 전문화**
- **실무에서 통하는 패턴 3가지**: Pipeline, Fan-Out, Specialist Team
- **에이전트 간 데이터 포맷 표준화**가 가장 중요한 기술적 결정
- **Dreaming 기능**으로 에이전트 팀이 경험에서 스스로 학습 (Harvey는 완료율 6배 향상)
- **2개짜리 파이프라인부터** 시작해서 점진적으로 확장하는 것이 핵심

## 본문

### 왜 멀티에이전트가 단일 에이전트를 이기나

단일 에이전트는 유능한 직원 한 명과 같다. 아무리 뛰어나도 한 번에 하나만 처리한다. 5가지 작업이 있으면 순서대로다.

멀티에이전트는 팀이다. 각기 다른 부분을 전문으로 하는 5명이 동시에 달린다. 단일 에이전트가 30분 걸리는 일을 5개 팀은 6분에 끝낸다.

그런데 속도보다 더 중요한 게 있다. **전문화**다. 리서치·분석·글쓰기·코딩·검토를 혼자 다 하는 에이전트는 모든 면에서 평범하다. 각 역할에 특화된 에이전트 팀은 모든 면에서 뛰어나다. 인간 팀이 개인을 이기는 이유와 똑같다.

### 실무에서 통하는 3가지 패턴

**1. 파이프라인 (Pipeline)**

에이전트가 순서대로 작동하며 출력을 다음 에이전트로 넘긴다.

`리서치 → 분석 → 작성 → 검토`

각 단계가 명확한 입출력을 가질 때 효과적이다. 리서치 에이전트가 데이터를 찾고, 분석 에이전트가 패턴을 발견하고, 작성 에이전트가 리포트를 만들고, 검토 에이전트가 오류를 잡는다.

**2. 팬아웃 (Fan-Out)**

커맨더 에이전트가 큰 작업을 쪼개 여러 워커 에이전트에 동시 분배한다. Netflix의 빌드 로그 분석이 바로 이 방식이다. 같은 작업을 다수의 대상에 독립적으로 적용할 때 최적이다.

**3. 스페셜리스트 팀 (Specialist Team)**

서로 다른 전문성을 가진 에이전트들이 하나의 복잡한 작업을 협력해 처리한다. Harvey의 법률 업무가 대표 사례다 — 리서치, 판례 분석, 문서 작성, 컴플라이언스 에이전트가 각 영역을 담당하고 결과를 종합한다.

### 7단계 구축 순서

**1단계 — 팀 정의**
전체 목표를 정하고, 하위 작업을 쪼개고, 병렬 가능한 것과 순서가 필요한 것을 구분한다. 각 하위 작업에 "어떤 전문가를 채용할까"를 생각하면 에이전트 역할이 나온다.

**2단계 — 에이전트 설계**
각 에이전트는 3가지가 필요하다: 명확한 역할, 필요한 특정 도구, 정해진 출력 형식. 출력 형식이 핵심이다. 에이전트 간 소통은 이 포맷으로 이루어진다.

**3단계 — 오케스트레이션 구성**
병렬로 실행할 에이전트와 순차로 실행할 에이전트를 결정한다. 에이전트가 실패했을 때의 폴백 동작도 반드시 정의한다. 하나가 실패해도 전체가 멈추지 않아야 한다.

**4단계 — Dreaming 활성화**
세션 사이에 돌아가는 백그라운드 프로세스다. 지난 세션을 검토해 패턴을 추출하고 실수를 파악해 메모리에 반영한다. 이걸 켜면 에이전트 팀이 시간이 지날수록 스스로 똑똑해진다. Harvey는 이것만으로 완료율이 6배 올랐다.

**5단계 — Outcomes 정의**
루브릭 기반 품질 기준을 설정한다. Claude가 기준에 맞게 자체 평가하고 통과할 때까지 반복한다. "충분히 좋은 게 뭔지"를 에이전트가 알게 된다.

**6단계 — 단계적 테스트**
에이전트 10개짜리로 처음부터 시작하지 말자. 2개짜리 간단한 파이프라인부터 시작해서, 커뮤니케이션·포맷·에러 처리가 완벽해지면 하나씩 추가한다.

**7단계 — 모니터링**
멀티에이전트는 단일 에이전트보다 훨씬 많은 것이 잘못될 수 있다. 핸드오프 실패, 중복 작업, 품질 저하, 토큰 낭비 4가지를 집중적으로 관찰한다.

### 실제 프로덕션 예시: 주간 시장 인텔리전스 리포트

- **Agent 1** 웹 리서치 → 최근 뉴스·제품 출시·투자 동향 수집 (병렬 실행)
- **Agent 2** 경쟁사 모니터 → 5개 경쟁사 웹사이트의 가격·기능 변화 추적 (병렬 실행)
- **Agent 3** 소셜 청취 → X와 LinkedIn의 관련 토론·트렌드 스캔 (병렬 실행)
- **Agent 4** 분석 → 1~3에서 받은 데이터로 상위 5개 주요 변화 선별 및 영향도 평가
- **Agent 5** 리포트 작성 → 분석 결과를 바탕으로 임원용 브리핑 생성
- **Agent 6** 품질 검토 → 루브릭 기준으로 검토, 수정 요청

단일 에이전트로 1시간 이상 걸리던 작업이 15분 이내로 줄었다. 수작업으로 반나절이던 일이.

## 실무자가 볼 핵심 포인트

1. **지금 당장 2개짜리 파이프라인부터 시작하라** — 생각보다 훨씬 단순하다, 이번 주 안에 할 수 있다
2. **에이전트 간 데이터 계약을 먼저 정의하라** — 어떤 필드, 어떤 포맷, 빈 값 처리까지
3. **Dreaming을 켜두면 팀이 스스로 학습한다** — 프롬프트를 손댈 필요가 점점 줄어든다
4. **좁은 역할이 강하다** — 범용 에이전트를 만들면 멀티에이전트의 의미가 없다
5. **에러 처리는 선택이 아니다** — 한 에이전트가 실패해도 전체가 계속 돌아야 한다
