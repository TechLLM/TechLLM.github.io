## 개요

구글 클라우드(Google Cloud)가 6월 14일(현지 시각) Open Knowledge Format, 줄여서 **OKF v0.1**을 공개했습니다. 한마디로 정리하면 *"사내 지식을 마크다운(Markdown) 한 묶음으로 표준화해서, 어떤 AI 에이전트든 그 묶음만 보면 회사 맥락을 이해하게 만들자"* 는 미니멀 스펙입니다.

요즘 코딩 에이전트를 써 본 분이라면 한 번쯤 겪었을 겁니다. 모델은 똑똑한데 *우리 회사*의 테이블 이름, 컨벤션, 정책을 모릅니다. 그래서 매번 같은 맥락을 프롬프트에 다시 붙여 넣게 됩니다. OKF는 바로 그 문제를 *포맷 표준*으로 풀어 보겠다는 시도입니다.

## 핵심 요약

- OKF v0.1은 사내 지식을 **YAML 프론트매터 + 마크다운 본문** 한 폴더로 묶는 포맷입니다.
- 필수 필드는 단 하나, `type`뿐입니다. 나머지는 생산자가 자유롭게 확장합니다.
- 문서끼리는 일반 마크다운 링크로 이어져, 자연스럽게 *지식 그래프*가 됩니다.
- 안드레이 카파시가 띄운 **"LLM 위키"** 패턴을 시스템 간 이식 가능한 형태로 끌어올린 결과물입니다.
- BigQuery 메타 추출 에이전트, HTML 시각화기, 샘플 번들 3종이 함께 공개됐고 GitHub에서 바로 받을 수 있습니다.

## 흩어진 지식이 AI 에이전트를 막아 세운다

{{< figure src="/images/google-cloud-okf-markdown-ai-agents/google-cloud-okf-markdown-ai-agents-body-01.png" alt="흩어진 사내 문서가 한쪽에 쌓여 있고 반대쪽에서는 같은 내용이 한 묶음의 정돈된 마크다운으로 다시 정리되는 모습을 표현한 유화풍 일러스트" caption="흩어진 사내 지식을 한 묶음의 마크다운으로 다시 정리한다" >}}

OKF가 푸는 문제는 어떤 회사에서든 익숙합니다. 사내 지식은 한곳에 없습니다. 데이터 카탈로그의 메타정보, 위키 페이지, 코드 주석, 노트북 셀, 그리고 결국 *담당자 머릿속*까지. AI 에이전트가 "이 데이터셋으로 SQL 짜 줘"라는 요청을 받으면, 이 조각들을 직접 긁어모아 맥락을 다시 짜야 합니다.

구글 클라우드는 이 상황을 두 줄로 정리합니다. 에이전트 개발자는 모두 같은 컨텍스트 문제를 *처음부터* 다시 푼다. 그리고 카탈로그 벤더는 모두 같은 데이터 모델을 *처음부터* 다시 만든다.

코딩 에이전트에 연결한 옵시디언 볼트, `AGENTS.md`나 `CLAUDE.md` 같은 컨벤션 파일, 데이터 팀의 "metadata as code" 저장소 — 모두 비슷한 패턴을 따릅니다. 하지만 각자 자기 시스템 안에서만 통합니다. 한 시스템에 들어간 지식은 다른 시스템으로 잘 넘어가지 못합니다. OKF는 바로 그 *벽*을 깨는 데 초점을 둡니다.

## 스펙은 최소한, 확장은 자유

OKF가 강요하는 건 거의 없습니다. 번들은 마크다운 파일들이 모인 디렉터리 한 개입니다. 각 파일은 YAML 프론트매터로 시작합니다. 거기서 **필수는 `type` 하나**뿐입니다. `title`, `description`, `resource`, `tags`, 타임스탬프 같은 필드는 권장은 되지만 선택입니다. 본문은 그냥 마크다운입니다.

문서끼리의 연결도 일반 마크다운 링크로 합니다. 거기서 *지식 그래프*가 자연스럽게 만들어집니다. 별도 그래프 DB가 필요 없습니다. 결과적으로 OKF 번들은 그 자체로 IDE에서 열리고, GitHub에서 바로 렌더링되고, 어떤 검색 도구로도 색인할 수 있습니다.

이 단순함이 핵심입니다. 어떤 `type`을 쓸지, 어떤 추가 필드를 넣을지, 본문을 어떻게 구성할지는 모두 생산자 마음입니다. 사람이 직접 손으로 쓴 번들도, 에이전트가 자동으로 뽑아낸 번들도 같은 포맷을 따릅니다. 소비자 입장에서는 둘이 구분되지 않습니다.

## 카파시의 "LLM 위키"를 표준 포맷으로

이 흐름의 출발점이 안드레이 카파시가 최근 주목받게 만든 **"LLM 위키"** 아이디어입니다. 회사 안의 지식을 마크다운 한 묶음으로 정리해 두면, 인간도 읽고 LLM도 읽을 수 있다는 단순한 주장이었습니다. 옵시디언, GitHub 리포지토리, `AGENTS.md` 패턴 모두 사실상 같은 발상의 변주입니다.

OKF는 이 패턴에 *공식 옷*을 입힌 것에 가깝습니다. 누가 만들었든, 어디에 저장됐든, 같은 규칙을 따르면 다른 시스템에서도 그대로 읽힌다는 *이식성*이 본질입니다. 클라우드, 데이터베이스, 에이전트 프레임워크가 무엇인지 따지지 않습니다.

## 같이 공개된 레퍼런스 구현

{{< figure src="/images/google-cloud-okf-markdown-ai-agents/google-cloud-okf-markdown-ai-agents-body-02.png" alt="작은 로봇이 빛나는 데이터 서버 옆에서 BigQuery 데이터셋을 끌어내 OKF 마크다운 문서로 쌓아 올리는 모습을 그린 유화풍 일러스트" caption="BigQuery 메타 추출 에이전트가 테이블마다 OKF 문서를 생성한다" >}}

구글 클라우드는 스펙만 던지지 않았습니다. 실제로 만져 볼 수 있도록 몇 가지 레퍼런스도 같이 풀었습니다.

- **BigQuery 메타 추출 에이전트**: 데이터셋을 크롤링하면서 테이블마다 OKF 문서를 자동으로 생성합니다.
- **정적 HTML 시각화기**: OKF 번들을 그대로 웹에서 둘러볼 수 있는 단순한 뷰어입니다.
- **샘플 번들 3종**: GA4 e-commerce, Stack Overflow, 비트코인 데이터셋이 포함됐습니다.

여기에 구글의 Knowledge Catalog도 업데이트돼, OKF 번들을 그대로 흡수해 에이전트에게 서빙할 수 있게 됐습니다. 스펙과 코드는 GitHub에 올라와 있고, Knowledge Catalog 연동은 별도 문서로 제공됩니다.

## 실무자가 볼 핵심 포인트

- **에이전트 컨텍스트 비용을 포맷으로 흡수합니다.** 같은 회사 정보를 매번 프롬프트로 옮기지 말고, OKF 번들 한 묶음을 만들어 두면 다음 에이전트도 그대로 씁니다.
- **`AGENTS.md`·`CLAUDE.md`를 쓰던 팀이 가장 자연스럽게 올라탑니다.** 이미 마크다운으로 컨벤션을 정리해 두고 있다면, 프론트매터만 OKF 규격으로 맞추면 됩니다.
- **데이터 팀은 BigQuery 에이전트가 좋은 출발점입니다.** 테이블별 자동 추출본을 만들고, 부족한 비즈니스 맥락만 사람이 덧붙이는 식으로 가면 비용이 낮습니다.
- **벤더 종속을 한 단계 풉니다.** 한 카탈로그에 묶여 있던 메타가 OKF로 빠지면, 다른 카탈로그·에이전트로 옮기는 비용이 크게 줄어듭니다.
- **그래프 DB를 미리 도입할 필요가 없습니다.** 마크다운 링크가 곧 그래프 엣지이므로, 일단 번들부터 만들고 시각화는 나중에 붙여도 됩니다.

## 마치며

OKF는 화려한 스펙이 아닙니다. 오히려 *덜 정한 것*이 매력입니다. 필수 필드 하나, 마크다운 본문, 그리고 마크다운 링크. 이 정도 약속만 모두가 지키면, 사내 위키와 데이터 카탈로그와 코딩 에이전트가 같은 언어로 이야기할 수 있다는 게 구글의 베팅입니다. 사내 지식을 AI에 어떻게 먹일지 고민하던 팀이라면, 일단 `AGENTS.md` 한 장부터 OKF 규격으로 옮겨 보는 데서 시작해도 충분합니다. 작게 시작해도 효과는 누적됩니다.
