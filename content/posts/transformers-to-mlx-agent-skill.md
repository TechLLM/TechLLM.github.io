에이전트가 코드를 짜고, 이슈를 찾고, PR까지 열어준다. 좋아 보인다. 그런데 오픈소스 메인테이너들은 지금 곤혹스럽다. PR이 10배 늘었는데, 대부분 쓸 수 없는 것들이다.

HuggingFace가 이 문제를 정면으로 파고든 글을 2026년 4월 공개했다. transformers 모델을 Apple Silicon용 MLX 프레임워크로 포팅하는 **AI 스킬**을 만들면서, 에이전트 시대 오픈소스 기여가 어때야 하는지 기준을 제시했다.

---

## 핵심 요약

- 코드 에이전트가 실제로 동작하는 2026년, 오픈소스 PR 폭증 문제가 본격화됐다
- 에이전트 생성 PR의 공통 실패 요인: 라이브러리 설계 철학 무시, 묵시적 계약 위반, 사소한 버그 다수
- HuggingFace는 **스킬(Skill)** — 에이전트를 위한 가이드라인 파일 — 을 만들어 포팅 품질을 표준화했다
- 스킬은 RoPE 버그, dtype 오염, 레이어별 수치 비교 등 숙련된 포터만 아는 체크를 자동화한다
- 별도의 **비-에이전틱 테스트 하네스**로 재현성과 투명성을 보장한다

---

## 에이전트가 PR을 망치는 이유

2026년에 코드 에이전트는 실제로 작동한다. 간단한 명세만 줘도 원하는 코드를 원샷으로 뽑아준다. Jensen Huang의 말처럼 "순식간에 코더가 3천만 명에서 10억 명으로 늘었다."

그런데 transformers 같은 핵심 라이브러리에는 문제가 생겼다. 누구나 에이전트에게 "오픈 이슈 찾아서 고치고 PR 열어줘"라고 시킨다. PR이 쏟아진다. 하지만 대부분 받아들이기 어렵다.

두 가지 전제를 에이전트가 놓친다.

**첫째, transformers는 코드 자체를 중시한다.** 모델 파일은 위에서 아래로 읽힌다. 복잡한 추상화 없이. 수천 명이 사용하는 라이브러리이기 때문에, 코드는 사람 간 소통 수단이다. 이 철학이 라이브러리 설계 전반에 스며있다.

**둘째, 에이전트는 그 맥락이 없다.** 그러니 "베스트 프랙티스"를 따라 "개선"을 시도하다 묵시적 계약을 깨뜨린다. 리팩터링, 이른 추상화, 다른 영역까지 영향을 주는 변경. 게다가 사람이었다면 메인테이너가 한 줄 댓글로 잘랐을 아이디어도 에이전트는 기꺼이 따라간다.

메인테이너 수는 그대로인데 PR은 10배가 됐다.

---

## 스킬: 에이전트를 위한 레시피 파일

HuggingFace는 MLX 포팅 문제를 이 방식으로 풀었다. **스킬(Skill)** — 에이전트가 복잡한 작업을 일관성 있게 수행하도록 안내하는 텍스트 파일이다.

스킬은 자동화가 아니다. 가이드라인이다. 프롬프트와 반복으로 같은 결과를 낼 수 있지만, 스킬은 **일관성**을 제공한다. 모든 실행이 같은 절차를 따른다. 누가 어떻게 프롬프트하느냐에 관계없이.

### 컨트리뷰터를 위한 자동화

- Hub에서 관련 모델 변형 탐색 및 다운로드
- 설정 파일 diff로 모델 변형 간 파라미터 차이 파악
- transformers 코딩 스타일을 기반으로 MLX 구현 작성
- **RoPE 설정 검증** — 긴 시퀀스에서 성능이 무너지는 버그를 잡는다
- safetensors 메타데이터에서 dtype 추론 (config에 명시 안 된 경우)
- 레이어별 transformers ↔ MLX 수치 비교로 발산 지점 정확히 특정

### 리뷰어를 위한 투명성

스킬이 만든 PR은 에이전트 지원임을 명시한다. 그러면서도 숙련된 사람이 제출한 것처럼 보인다.

- mlx-lm 컨벤션 준수: 불필요한 주석 없음, 투기성 추상화 없음
- PR 본문에 포함: 모델 변형 요약, 생성 예시, 수치 비교, dtype 검증, 레이어별 비교
- 컨트리뷰터가 결과를 검토·승인해야만 PR 오픈

## 비-에이전틱 테스트 하네스

스킬의 결과물을 에이전트가 검사하면 한계가 있다. LLM이 결과를 환각하거나 너무 관대하게 평가할 수 있다. 그래서 **별도의 테스트 하네스**를 만들었다.

에이전트와 독립적으로 실행되는, 재현 가능한 테스트 시스템이다.

- **환각 제거**: LLM이 결과를 꾸미거나 적당히 넘어가지 않는다
- **재현성**: 누구나 레포를 받아서 같은 테스트를 돌릴 수 있다
- **투명성**: 요약 보고서, 모델별 상세 결과, 원시 입출력 JSON, 실행된 테스트 스크립트가 모두 저장된다

단, 테스트 하네스는 CI 게이트가 아니다. "사전학습 모델이 긴 시퀀스에서 반복하는 게 정상인가?", "transformers 대비 4% 로짓 차이는 허용 범위인가?" 같은 판단은 여전히 사람이 한다. 하네스는 신호를 제공하고, 판단은 리뷰어와 컨트리뷰터가 내린다.

---

## 실무자가 볼 핵심 포인트

1. **스킬은 자동화가 아닌 표준화다** — "AI가 알아서 PR 올린다"가 아니라 "AI가 숙련된 포터의 체크리스트를 따르도록 안내한다"는 개념이다. 목적이 다르다.

2. **에이전트 PR의 실패 패턴을 알아야 한다** — 묵시적 계약 위반, 불필요한 추상화, 관련 없는 영역 수정. 에이전트를 활용한다면 이 패턴을 명시적으로 막아야 한다.

3. **비-에이전틱 검증 레이어를 분리하라** — 에이전트가 테스트하고 에이전트가 결과를 확인하면 신뢰성이 낮다. HF처럼 독립된 재현 가능한 테스트 시스템을 별도로 두는 것이 맞다.

4. **컨트리뷰터는 코드를 소유해야 한다** — 스킬이 PR을 만들어줘도, 리뷰어 피드백을 다시 에이전트에게 넘기면 안 된다. 에이전트는 자기 결정을 고수하고 탈선한다. 리뷰어와의 대화는 사람 대 사람이어야 한다.

5. **설치와 실행은 단순하다** — `uv run https://...install_skill.py && uvx hf skills add --claude` 두 줄이면 된다. Claude Code 기반이며 Codex 등 다른 코딩 에이전트에도 이론적으로 적용 가능하다.
