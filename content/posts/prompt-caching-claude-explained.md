## 매 턴마다 지불하는 '세금'

AI 에이전트가 동작할 때, 모든 턴마다 숨은 비용이 쌓인다. 시스템 지침, 도구 정의, 이전에 로드한 프로젝트 컨텍스트까지 — 전부 처음부터 다시 읽는 것이다. 이것이 **컨텍스트 세금(Context Tax)**이다.

50턴에 걸쳐 20,000토큰의 시스템 프롬프트를 실행하면, 100만 토큰의 불필요한 계산이 발생한다. 새로 생긴 값 없이도.

이 글은 Akshay(@akshay_pachaar)의 트레드 분석을 바탕으로, **프롬프트 캐싱이 실제로 어떻게 작동하는지**, 그리고 **클로드 코드에서 92% 캐시 히트율과 81% 비용 절감이 어떻게 가능한지**를 설명한다.

## 변하는 것과 변하지 않는 것

에이전트 프롬프트(컨텍스트)는 근본적으로 다른 두 파트로 나뉜다:

**정적 접두사(Static Prefix)**
시스템 지침, 도구 정의, 프로젝트 컨텍스트, 행동 지침이 포함된다. 세션 내내 모든 턴에서 동일하다.

**동적 꼬리(Dynamic Tail)**
사용자 메시지, 도구 출력, 최종 관찰값. 매 요청마다 다르며 대화 진행과 함께 증가한다.

캐싱의 핵심은 **정적 접두사가 매번 불필요하게 재계산되는 비용을 없애는 것**이다. 한 번 처리하고, 이후 턴에서는 메모리에서 읽기만 한다.

## KV 캐싱은 왜 효과적인가

LLM 추론은 두 단계로 나뉜다:

**프리필(Prefill)**
모델이 전체 입력 프롬프트를 처리한다. 모든 토큰에 대해 밀집 행렬 곱셈을 실행하므로 **연산 집약적(compute-bound)**이다. 모든 것을 읽고 표현을 구축하는 느리고 비싼 단계다.

**디코드(Decode)**
출력 토큰을 하나씩 생성한다. **메모리 집약적(memory-bound)**으로, 이전에 계산된 상태를 읽는 데 대부분의 시간을 소요한다.

프리필 단계에서 트랜스포머는 각 토큰마다 세 개의 벡터를 생성한다:
- **Query(Q)**: 현재 토큰의 질의
- **Key(K)**: 토큰 간 관계를 파악하는 색인
- **Value(V)**: 실제 어텐션 값

핵심 통찰: **Key와 Value 벡터는 이전 토큰에만 의존한다.** 한 번 계산되면 절대 변하지 않는다.

캐싱 없으면: 매 요청 종료 시 KV 텐서가 버려지고, 다음 요청이 20,000토큰 모두 처음부터 재계산한다.

캐싱 있으면: 암호학적인 해시로 인덱싱하여 서버에 저장한다. 다음 요청 시 해시가 일치하면 즉시 텐서를 Retrieved하여 모든 재계산을 건너뛴다.

이로 인해 생성 토큰당 계산 복잡도가 **O(n²)에서 O(n)**으로 감소한다.

## 가격 구조의 실제

Anthropic의 캐시 가격 정책은 세 가지 숫자로 기억해야 한다:

| 操作 | 가격 |
|---|---|
| 캐시 읽기 | 기본 입력 가격의 **10%** (90% 할인) |
| 캐시 쓰기 | 기본 입력 가격의 **125%** (저장 프리미엄) |
| 1시간 연장 캐시 | 기본 가격의 **2배** |

수학이 작동하려면 캐시 히트율이 높아야 한다. 실제 데이터가 얼마나 맞아떨어지는지 보자.

## Claude Code: 30분 세션 워크스루

Claude Code는 **캐시를 따뜻하게 유지**하는 것 하나에 모든 것을 구축했다.

**0분: 세션 시작**
CLAUDE.md 파일과 시스템 프롬프트를 로드한다. 이 페이로드가 쉽게 20,000토큰을 초과한다. 세션에서 가장 비싼 순간이지만, 이 비용은 단 한 번만 지불한다.

**1~5분: 첫 번째 명령**
Explore 서브에이전트가 코드베이스를 탐색하고 파일을 열며 grep을 실행한다. 모든 출력이 동적 꼬리에 추가된다. 20,000토큰 정적 기반은 이미 캐시에 있다. $0.30/MTok로 읽히며, 새 도구 출력과 메시지만 새로 계산된다.

**6~15분: 딥워크**
Plan 서브에이전트가 Explore 결과를 받는다. 원본 결과를 그대로 전달하지 않고요약을 전달하여 동적 꼬리를 관리 가능한 상태로 유지한다. 구현 계획을 수립하고, 사용자가 승인하면 변경을 시작한다. 매 턴마다 20,000토큰 접두사를 캐시에서 읽는다. 각 캐시 히트는 TTL을 초기화하여 캐시를 따뜻하게 유지한다.

**16~25분: 반복**
수정 요청, 접근 방식 수정, 더 많은 도구 호출. 동적 꼬리가 성장하지만 오직 새로운 고유 콘텐츠만 새 계산이다. 이 시점에서 세션은 수십만 토큰을 처리했지만, 20,000토큰 기반은 매 턴 캐시에서 읽혔다.

**28분: /cost 실행**
캐싱 없으면 이런 세션은 쉽게 200만 토큰을 초과한다. Sonnet 4.5 기준 약 $6.00에 해당한다. 캐싱이 효율적으로 작동하면:대부분의 토큰이 $0.30/MTok로 캐시에서 읽히고, 동적 꼬리 토큰만 새로 계산된다. 실제 프로젝트에서 80%+ 비용 절감이 가능하다.

## 캐시를 무효화하는 가장 위험한 규칙

프롬프트 캐싱에서 가장 직관에 반하는 사실:

**1 + 2 = 3. But 2 + 1 is a cache miss.**

인프라가 프롬프트의 해시를 계산한다. 해시가 변하면(순서 하나만 바뀌어도) 캐시가 비고, 전체 접두사가 정가로 재계산된다.

이로부터 세 가지 규칙이 따른다:

1. **세션 중 도구를 추가하거나 제거하지 마라.** 캐시된 접두사에 도구가 포함된다. 도구 변경은 이후 모든 것을 무효화한다.
2. **세션 중 모델을 전환하지 마라.** 캐시는 모델-specific하다. 더 저렴한 모델로 전환하면 전체 캐시를 재구축해야 한다.
3. **상태 변경을 위해 접두사를 변경하지 마라.** 대신 다음 사용자 메시지에 태그를 추가하여 시스템에 알린다. 접두사는 절대 변하지 않는다.

## 실무자를 위한 구조화 원칙

직접 에이전트를 만든다면 프롬프트를 다음과 같이 구조화해야 한다:

1. **맨 위에 시스템 지침과 규칙** — 세션 중 변경 금지
2. **필요한 모든 도구를 미리 로드** — 추가/제거 금지
3. **그 다음 검색된 컨텍스트와 문서** — 세션 내내 정적
4. **맨 아래에 대화 기록과 도구 출력** — 동적 영역

Auto-caching이 켜져 있으면 브레이크포인트가 대화 진행에 따라 자동으로 앞으로 이동한다. Claude Code가 자신의 캐시를 관리한다. Anthropic이 API에 auto-caching을 추가했으니, 직접 만든 에이전트에도 같은 것을 적용할 수 있다.

## 세 가지 모니터링 지표

API가 제대로 작동하는지 확인하려면 응답의 세 가지 필드를 확인해야 한다:

- **cache_creation_input_tokens**: 메모리에 저장된 토큰 수
- **cache_read_input_tokens**: 메모리에서 읽은 토큰 수
- **input_tokens**: 평소대로 계산된 토큰 수

캐시 효율성 = 읽은 토큰 수 / 생성 토큰 수. 업타임처럼 이것도 주시해야 한다.

## 핵심 정리

프롬프트 캐싱은 그냥 켜는 기능이 아니라, **구조적으로 설계하는 원칙**이다.

클로드 코드가 그 모습을 보여준다. 92% 캐시 히트율, 81% 비용 절감. 에이전트를 만든다면 이 것이 블루프린트다.
