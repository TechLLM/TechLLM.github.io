## 핵심 요약

AI 에이전트를 만들 때 많은 팀이 먼저 모델과 도구를 고릅니다. 어떤 LLM을 쓸지, 어떤 프레임워크를 붙일지, 어떤 벡터 DB를 연결할지부터 고민합니다. 그런데 Oracle의 글 **“Oracle AI Agent Memory”**가 짚는 핵심은 조금 다릅니다.

프로덕션 에이전트에서 진짜 어려운 문제는 **“기억을 어디에, 어떻게, 누구 책임 아래 둘 것인가”**입니다.

## 메모리는 더 이상 부가 기능이 아니다

초기 에이전트 데모에서는 대화 기록과 벡터 검색 정도만 붙여도 그럴듯해 보입니다. 하지만 기업 환경으로 들어가면 질문이 달라집니다. 이 기억은 누가 소유하는가? 테넌트별로 안전하게 분리되는가? 감사 로그를 남길 수 있는가? 사용자가 삭제를 요청하면 지울 수 있는가?

컨텍스트 윈도우가 커졌다고 해도, 장기간 일하는 에이전트의 전체 상태를 매번 프롬프트에 넣을 수는 없습니다. 사용자 선호, 과거 작업 결과, 도메인 지식, 도구 실행 이력, 절차적 규칙은 시간이 지나며 계속 쌓입니다. 그래서 에이전트 메모리는 연구용 장난감이 아니라 운영 인프라가 됩니다.

## 네 가지 메모리를 하나의 코어로 본다

글은 에이전트 메모리를 네 가지로 나눕니다.

**Working memory**는 지금 추론에 쓰는 현재 대화와 작업 상태입니다. **Semantic memory**는 사용자 선호, 엔티티, 업무 지식 같은 오래 남는 사실입니다. **Episodic memory**는 과거 세션에서 실제로 무슨 일이 있었는지에 대한 경험 기록입니다. **Procedural memory**는 에이전트가 어떤 방식으로 행동해야 하는지 알려주는 규칙과 절차입니다.

중요한 건 이것들이 네 개의 별도 시스템이 아니라는 점입니다. 실제 운영에서는 같은 상태를 서로 다른 방식으로 읽고 쓰는 접근 패턴에 가깝습니다. 그래서 Oracle은 여러 서비스를 억지로 이어 붙이는 대신, Oracle AI Database 위에 통합 메모리 코어를 두는 방식을 제안합니다.

## Oracle이 내세우는 차별점

Oracle AI Agent Memory는 Python 패키지 `oracleagentmemory`로 제공되며, 프레임워크에 종속되지 않는 메모리 레이어를 지향합니다. LangGraph, Claude Agent SDK, OpenAI Agents SDK, WayFlow, 커스텀 Python 하네스에서 같은 메모리 클라이언트를 사용할 수 있다는 점을 강조합니다.

핵심은 벡터 검색만이 아닙니다. Oracle AI Database가 제공하는 관계형 쿼리, 벡터 검색, JSON 문서 처리, 트랜잭션 일관성, 암호화, 접근 제어, 감사 기능을 한 저장소에서 묶는 데 있습니다. 기업 입장에서는 벡터 DB, 채팅 로그, 추출 스크립트, 권한 관리 시스템을 각각 따로 운영하는 부담을 줄일 수 있습니다.

## 숫자로 본 효과

Oracle은 LongMemEval 벤치마크에서 93.8% 점수를 기록했다고 밝힙니다. 단일 세션 회상은 100%, 시간 추론은 96%, 지식 업데이트는 95%, 가장 어려운 멀티 세션 회상도 88%라고 합니다.

비용 측면도 흥미롭습니다. 80턴 대화 실험에서 전체 히스토리를 계속 넣는 방식은 마지막 요청 기준 약 13,900 토큰까지 늘었지만, Oracle AI Agent Memory는 요청당 입력을 약 1,300 토큰 수준으로 유지했다고 합니다. 긴 대화를 다루는 에이전트라면 비용 구조 자체가 달라지는 셈입니다.

## 실무자가 볼 포인트

이 글의 메시지는 분명합니다. 엔터프라이즈 에이전트에서 메모리는 “검색하면 좋은 부가 기능”이 아니라 **거버넌스가 필요한 핵심 상태 저장소**입니다.

에이전트가 무엇을 배웠는지, 어떤 사용자에게 속한 기억인지, 언제 삭제할 수 있는지, 어떤 프레임워크가 읽고 쓰는지까지 관리되어야 합니다. 모델이 좋아질수록 오히려 이런 기반 구조의 중요성은 더 커집니다. 똑똑한 에이전트일수록 더 많은 상태를 만들고, 그 상태가 잘못 관리되면 더 큰 리스크가 되기 때문입니다.

결국 에이전트 메모리의 기준은 “잘 검색되는가”에서 “안전하게 기억하고, 필요한 만큼만 꺼내며, 지워야 할 때 지울 수 있는가”로 이동하고 있습니다. 겅허게 말하면, 기업용 AI 에이전트의 승부는 프롬프트보다 메모리 운영에서 갈릴 가능성이 큽니다.
